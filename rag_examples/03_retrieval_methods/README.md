## 📖 学习指导

### 为什么检索是 RAG 最关键的一步？

```
检索不准 → 给 LLM 的信息都是错的 → 回答再强也没用（垃圾进，垃圾出）
检索精准 → 给 LLM 的信息都是对的 → 回答质量自然高

类比：考试
  检索 = 在教科书里找到相关章节
  生成 = 基于找到的内容组织答案
  如果找错章节，组织得再好也答非所问
```

### 第 1 步：标量查询（01_scalar_query.py）

**本节讲什么：**
- 什么是标量字段（INT64、FLOAT、VARCHAR、BOOL）vs 向量字段
- 单条件过滤：等于、大于、小于、布尔值
- 组合条件：AND / OR / IN / 括号嵌套
- 标量查询 + 向量检索的结合使用

**核心概念：**

```
标量字段 = 传统数据库中的"列"
  category = "AI"        ← VARCHAR 字段
  views = 1000           ← INT64 字段
  price = 99.0           ← FLOAT 字段
  is_published = True    ← BOOL 字段

向量字段 = Embedding 生成的"数字指纹"
  embedding = [0.12, -0.45, 0.78, ...]  ← FLOAT_VECTOR 字段
```

**过滤表达式语法（与 SQL 类似）：**

```python
# 单条件
"category == 'AI'"
"views > 800"
"is_published == True"

# 组合条件
"category == 'AI' and views > 600"
"category == 'AI' or category == 'Product'"
"category in ['AI', 'Data']"
"is_published == True and (views > 500 or price == 0)"

# 范围查询
"views >= 500 and views <= 1000"
"price < 100"
```

**标量 + 向量混合使用：**

```python
# 在向量检索的同时进行标量过滤
results = client.search(
    collection_name="knowledge_base",
    data=[query_vector],             # 语义检索：找语义相似的
    filter="category == 'AI'",       # 标量过滤：限定类别
    limit=5,
    output_fields=["title", "content"]
)
# 结果：只返回 AI 类别中最相似的 5 条
```

**典型 RAG 场景：**

| 用户需求 | 向量检索 | 标量过滤 | 组合效果 |
|---------|---------|---------|---------|
| "AI 相关的付费教程" | 找"AI 教程"相关内容 | `category == 'AI' and price > 0` | 精确筛选 |
| "最近 7 天的文章" | 找相关问题内容 | `publish_date >= '2024-01-01'` | 时间过滤 |
| "只读已发布的内容" | 找相关内容 | `is_published == True` | 状态过滤 |

### 第 2 步：向量检索（02_vector_search.py）

**本节讲什么：**
- 向量检索的完整流程：问题 → Embedding → 向量检索 → Top-K 结果
- 3 种度量类型的对比（COSINE / L2 / IP）
- 批量向量检索（一次传入多个查询向量）
- 检索参数详解（limit / output_fields / filter / search_params）
- 真实 Embedding 模型的使用方式

**核心概念：**

```
向量检索流程：

用户问题："机器学习是什么"
      ↓ Embedding 模型
查询向量：[0.8, 0.8, ..., -0.1, ...]（1024 维）
      ↓ 向量检索
与库中每个向量计算相似度
      ↓ 排序
返回 Top-K 最相似的文档

关键：查询向量和库中向量必须用同一个 Embedding 模型生成！
     否则不在同一语义空间，相似度计算无意义
```

**3 种度量类型对比：**

| 类型 | 计算方式 | 判断标准 | 特点 | 何时用 |
|------|---------|---------|------|--------|
| **COSINE** | 余弦相似度 | 越大越相似（-1 ~ 1） | 不受向量长度影响 | **推荐默认** |
| L2 | 欧氏距离 | 越小越相似 | 考虑向量绝对位置 | 向量未归一化时 |
| IP | 内积 | 越大越相似 | 归一化后 = COSINE | 追求极致性能 |

**检索参数详解：**

```python
results = client.search(
    collection_name="knowledge_base",
    data=[query_vector],           # 查询向量列表（支持批量）
    limit=5,                       # 返回 Top-5 最相似
    output_fields=["content", "title"],  # 返回哪些字段
    filter="category == 'AI'",     # 可选：标量过滤条件
    search_params={"nprobe": 10}   # 可选：索引搜索参数
)
# 返回值：[[Hit1, Hit2, Hit3, Hit4, Hit5]]
# 每个 Hit：
#   hit['distance'] = 相似度分数
#   hit['entity']['content'] = 文档内容
#   hit['entity']['title'] = 文档标题
```

**相似度阈值建议：**

```python
# 过滤掉不相关的结果
for hit in results[0]:
    if hit['distance'] > 0.5:   # COSINE 场景，0.5-0.7 是合理阈值
        contexts.append(hit['entity']['content'])
```

### 第 3 步：关键字检索（03_keyword_search.py）

**本节讲什么：**
- 最简单的关键词匹配（字面包含）
- BM25 算法原理（TF + IDF + 文档长度归一化）
- 从零实现 BM25（教学版）
- 使用 `rank-bm25` + `jieba` 进行中文 BM25 检索
- 关键字检索 vs 向量检索的对比

**核心概念：**

```
BM25 算法三要素：

1. TF（词频）- Term Frequency
   "机器学习"在文档中出现越多 → 分数越高
   但有边际效应：出现 10 次 ≠ 出现 2 次的 5 倍重要

2. IDF（逆文档频率）- Inverse Document Frequency
   "机器学习"在越少文档中出现 → 分数越高
   "的"、"是"等常见词 IDF 很低（没什么区分度）

3. 文档长度归一化
   长文档天然词频高 → 需要打折
   b=0：不考虑长度；b=1：完全归一化

BM25 公式（简化）：
  Score = Σ [ IDF(词) × TF(词, 文档) ]
```

**关键字 vs 向量检索对比：**

```
文档库：
  [1] "机器学习是人工智能的核心技术，通过数据训练模型。"
  [2] "深度学习使用神经网络，是 ML 的重要分支。"
  [3] "AI 模型通过训练可以自动学习规律和模式。"

用户查询："AI 学习"

关键字检索结果：
  → 查找包含"A"、"I"、"学"、"习"字符的文档
  → 匹配到所有包含这些字的文档（包括不相关的）
  → ✗ 无法理解"AI"="人工智能"，"学习"="训练"

向量检索结果：
  → 理解"AI 学习"的语义
  → 找到语义相关的文档 [1] 和 [3]
  → ✓ 能找到同义词相关但字面不同的内容
  → ✗ 但对专有名词（如"Milvus 2.4 版本"）可能匹配不精确

结论：两种方法互补，不是替代关系
```

**BM25 参数建议：**

| 参数 | 推荐值 | 含义 |
|------|--------|------|
| k1 | 1.2-2.0（默认 1.5） | 词频饱和度，控制 TF 边际效应 |
| b | 0.5-0.8（默认 0.75） | 长度归一化，0=不考虑长度 |

**中文分词工具选择：**

| 工具 | 特点 | 适用 |
|------|------|------|
| jieba | 轻量级，通用场景 | **推荐默认** |
| HanLP | 功能丰富，支持实体识别 | 需要识别人名/地名 |
| THULAC | 清华出品，学术场景 | NLP 研究 |

### 第 4 步：混合检索（04_hybrid_search.py）

**本节讲什么：**
- 为什么需要混合检索（向量 + 关键字互补）
- 两种结果融合方法：加权平均 vs RRF（倒数排名融合）
- 封装 HybridSearcher 类（向量检索 + BM25 + 融合）
- Milvus 原生混合检索 API 介绍

**核心概念：**

```
为什么需要混合检索？

  向量检索：理解语义，但专有名词匹配不精确
  关键字检索：精确匹配关键词，但不理解同义词

  两者结合 = 既理解语义，又保证专有名词精确匹配

混合检索流程：
  用户查询
     ↓
  ┌──────┬──────┐
  │向量检索│关键字检索│
  │(语义) │(精确) │
  └──┬───┴───┬──┘
     ↓       ↓
  结果 A   结果 B
     ↓       ↓
  ┌──┴───────┴──┐
  │  结果融合     │  ← RRF 或 加权平均
  └──────┬──────┘
         ↓
     最终结果
```

**两种融合方法对比：**

**方法 1：加权平均**
```python
fused_score = α × vector_score + (1-α) × keyword_score
# α = 0.5：两者平等
# α = 0.7：更侧重语义
# α = 0.3：更侧重关键字
```

**方法 2：RRF（倒数排名融合）— 推荐**
```python
# 公式：score = 1/(k + rank_vector) + 1/(k + rank_keyword)
# k = 60（默认值）
# 只看排名，不看原始分数 → 不同检索方法的分数不可比，但排名可比

# 例子：
# 文档A：向量排名1，关键字排名3 → RRF = 1/61 + 1/63 = 0.0323
# 文档B：向量排名2，关键字排名2 → RRF = 1/62 + 1/62 = 0.0323
# 文档C：向量排名5，关键字排名1 → RRF = 1/65 + 1/61 = 0.0318
```

**RRF vs 加权平均：**

| 维度 | RRF | 加权平均 |
|------|-----|---------|
| 依赖原始分数 | ✗ 只看排名 | ✓ 需要归一化分数 |
| 分数可比性 | ✓ 排名总是可比 | ✗ 不同方法分数可能不可比 |
| 调参 | k=60 通用 | 需要反复调 α |
| 稳定性 | ✓ 更稳定 | 需要经验 |
| 推荐度 | **首选** | 备用 |

**权重设置建议：**

| 场景 | 向量权重 | 关键字权重 | 原因 |
|------|---------|-----------|------|
| 通用 | 0.5 | 0.5 | 平衡 |
| 语义理解为主 | 0.7 | 0.3 | 理解用户意图 |
| 精确匹配为主 | 0.3 | 0.7 | 找专有名词/版本号 |

### 第 5 步：Rerank 重排序（05_rerank.py）

**本节讲什么：**
- 两阶段检索架构：召回（快）→ 精排（准）
- Bi-Encoder vs CrossEncoder 的精度与速度权衡
- BGE-Reranker 和 CrossEncoder 的实际使用
- Rerank 前后的效果对比（NDCG 提升）
- 完整的 RAG + Rerank 流程

**核心概念：**

```
两阶段检索架构：

阶段 1：召回（Recall）— 快
  向量检索 + 关键字检索 → Top-50 候选
  模型：Bi-Encoder（将 query 和 doc 分别编码再比较）
  速度：~10ms

阶段 2：精排（Rerank）— 准
  对 Top-50 逐一与 query 比较 → Top-5 最终
  模型：CrossEncoder（将 query + doc 一起编码）
  速度：~100-500ms

为什么需要两阶段？
  CrossEncoder 最准，但太慢 → 不能对百万文档逐一计算
  先用 Bi-Encoder 快速筛出 50 个候选
  再用 CrossEncoder 精细排序 → 既快又准
```

**Bi-Encoder vs CrossEncoder：**

```
Bi-Encoder（用于召回阶段）：
  query → [Encoder] → 向量Q
  doc   → [Encoder] → 向量D
  相似度 = cosine(向量Q, 向量D)
  ✓ 预计算所有文档向量 → 检索极快
  ✗ query 和 doc 独立编码 → 无法交互

CrossEncoder（用于精排阶段）：
  [query + doc] → [Encoder] → 相关分数
  ✓ query 和 doc 一起编码 → 充分交互，更准
  ✗ 每对都要计算 → 慢
```

**Rerank 效果对比：**

```
用户查询："机器学习需要什么基础？"

召回结果（按向量相似度排序）：
  [1] 0.85 - 深度学习是机器学习的子集，使用神经网络。
  [2] 0.82 - 机器学习通过训练数据让计算机自动学习。
  [3] 0.78 - Python 是常用的编程语言，用于 AI 开发。
  [4] 0.75 - 数据结构和算法是编程的基础。
  [5] 0.72 - 机器学习需要数学基础，包括线性代数和概率。

Rerank 后结果：
  [1] 0.92 ↑ - 机器学习需要数学基础，包括线性代数和概率。  ← 直接回答问题！
  [2] 0.68 = - 机器学习通过训练数据让计算机自动学习。
  [3] 0.45 ↓ - 深度学习是机器学习的子集，使用神经网络。
  [4] 0.32 ↓ - Python 是常用的编程语言，用于 AI 开发。
  [5] 0.28 ↓ - 数据结构和算法是编程的基础。

观察：第 5 条从最后一名升到第一名 — 因为它直接回答了"需要什么基础"
```

### 5 种检索方法全景图

```
                    检索方法
                   /    |    \
              标量   向量   关键字
              查询   检索   检索
               │      │      │
          条件筛选   语义匹配  字面匹配
          (AND/OR)  (COSINE)  (BM25)
                      \    /
                     混合检索
                    /        \
                 RRF        加权平均
                   │
              Rerank 重排序
              /           \
         CrossEncoder    BGE-Reranker
```

### 检索方法对比速查表

| 方法 | 原理 | 优点 | 缺点 | 何时用 |
|------|------|------|------|--------|
| 标量查询 | SQL 条件筛选 | 精确、快速 | 无法语义匹配 | 有明确过滤条件 |
| 向量检索 | 向量相似度 | 理解语义、同义词 | 专有名词不精确 | 语义搜索（默认） |
| 关键字检索 | BM25 算法 | 精确匹配关键词 | 不理解同义词 | 专有名词/版本号 |
| 混合检索 | 向量 + BM25 | 综合优势 | 实现复杂 | **生产环境首选** |
| Rerank | CrossEncoder 精排 | 精度最高 | 增加延迟 | 高精度要求 |

### 检索在 RAG 中的位置

```
知识库构建：
  加载文档 → 切片 → Embedding → 存入 Milvus

检索问答：
  用户提问 → Embedding → 【向量检索】→ Top-K 文档 → LLM 生成答案
                                 ↑
                          本节学的核心内容
```

## ⚠️ 常见问题

### Q: 向量检索返回的结果不相关怎么办？
A: 检查 3 点：① 查询向量和库中向量是否用同一个 Embedding 模型生成；② 维度是否匹配（都应该是 1024 维）；③ 度量类型是否正确（推荐 COSINE）。

### Q: 标量查询的字符串过滤不生效？
A: 注意字符串区分大小写，且需要单引号包裹：`filter="category == 'AI'"`，不要用双引号包裹字符串值。

### Q: BM25 的中文分词为什么用 jieba？
A: jieba 是最常用的中文分词库，轻量且效果好。BM25 需要对中文正确分词才能统计词频，直接按字符切分效果很差。

### Q: 什么时候需要 Rerank？
A: 对精度要求高的场景（如客服问答、法律/医疗检索）。如果你的 Top-5 向量检索结果已经够用，可以不加分。Rerank 会增加 100-500ms 延迟。

### Q: 混合检索中 RRF 的 k 值怎么选？
A: k=60 是论文推荐的默认值，适用于大多数场景。k 小（20-40）会让排名靠前的优势更大；k 大（80-100）会让两种方法权重更接近。
