## 📖 学习指导

### 为什么 RAG 系统需要评估？

```
不评估 = 盲目飞行：

换了 Embedding 模型 → 效果变好还是变差？不知道
调整了切片参数     → 检索质量有没有提升？不知道
切换了 LLM         → 回答质量有没有变化？不知道
更新了 100 篇文档   → 新文档被正确检索到了吗？不知道

评估的价值：
  ✓ 量化改进：知道每次改动到底有没有效果
  ✓ 发现问题：定位检索器或生成器的短板
  ✓ 版本对比：A/B 测试不同方案
  ✓ 上线信心：有数据支撑的质量保证
```

### 第 1 部分：RAGAS 四大核心指标

**本节讲什么：**
- RAGAS 框架（Retrieval Augmented Generation Assessment）
- 4 个指标分为两类：检索质量（精确率、召回率）和生成质量（忠实度、答案相关性）
- 每个指标的含义、计算公式、生活化比喻

**四大指标详解：**

**🔵 检索质量指标（评估检索器）：**

| 指标 | 公式 | 含义 | 生活化比喻 |
|------|------|------|-----------|
| **Context Precision（精确率）** | 相关文档数 / 检索到的文档总数 | 检索到的文档中有多少是真正相关的？ | Google 搜索结果前 10 条有几条有用？ |
| **Context Recall（召回率）** | 检索到的相关文档数 / 所有相关文档总数 | 所有相关文档中，有多少被检索到了？ | 图书馆里的相关书你找到了几本？ |

**🟢 生成质量指标（评估 LLM）：**

| 指标 | 含义 | 目标 | 生活化比喻 |
|------|------|------|-----------|
| **Faithfulness（忠实度）** | 回答是否完全基于检索到的上下文 | 不应出现上下文中没有的信息（幻觉） | 考试时你是否只用了给出的参考资料？ |
| **Answer Relevancy（答案相关性）** | 回答是否完整、准确地回答了用户问题 | 不能答非所问，不能遗漏关键信息 | 你问"今天天气怎么样"，回答"下雨"而不是"今天是星期五" |

**四个指标在 RAG 流程中的位置：**

```
用户问题
   ↓
[检索器] ──→ Context Precision + Context Recall（检索质量）
   ↓
检索上下文 + 问题
   ↓
[LLM 生成] ──→ Faithfulness + Answer Relevancy（生成质量）
   ↓
最终答案
```

**精确率 vs 召回率的权衡：**

```
高精确率 + 低召回率：
  检索到的都很相关，但漏掉了很多相关内容
  → LLM 收到的信息少，可能回答不完整

低精确率 + 高召回率：
  相关内容基本都找到了，但混入了很多噪音
  → LLM 被无关信息干扰，可能产生幻觉

理想状态：高精确率 + 高召回率
  → 相关的全找到了，不相关的都没出现
```

### 第 2 部分：构建评估数据集

**本节讲什么：**
- 评估数据集的结构：`question` + `ground_truth` + `relevant_docs`
- 4 条构建原则（覆盖知识点、各种题型、数量建议、标注要点）

**评估数据的结构：**

```json
[
  {
    "question": "什么是 RAG？",
    "ground_truth": "RAG 是检索增强生成，结合检索和生成的技术...",
    "relevant_docs": ["rag_intro.txt", "rag_guide.txt"]
  },
  {
    "question": "Milvus 支持哪些检索方式？",
    "ground_truth": "Milvus 支持标量查询、稠密向量检索、BM25 和混合检索。",
    "relevant_docs": ["milvus_doc.txt", "search_guide.txt"]
  }
]
```

**4 条构建原则：**

| 原则 | 说明 | 示例 |
|------|------|------|
| 覆盖主要知识点 | 不要所有问题都问同一个概念 | 覆盖文档的不同章节/主题 |
| 各种题型混合 | 事实性、对比性、应用性 | "是什么"、"A 和 B 区别"、"用在什么场景" |
| 数量建议 | 最少 20 条，推荐 50-100 条，生产 200+ 条 | 太少没有统计意义 |
| 标注要点 | ground_truth 简洁完整，relevant_docs 包含所有相关文档 | 可加入"无关文档"测试抗噪能力 |

### 第 3 部分：RAGEvaluator 评估器

**本节讲什么：**
- `RAGEvaluator` 类的完整实现（简化版 RAGAS）
- 4 个指标的计算方法
- `evaluate_all()` 批量评估流程
- 评估报告生成与评级

**核心类结构：**

        """添加一条评估用例"""

    # ── 四个指标计算 ──

    def context_precision(retrieved_docs, relevant_docs):
        """精确率 = |检索到且相关| / |检索到的|"""

    def context_recall(retrieved_docs, relevant_docs):
        """召回率 = |检索到且相关| / |所有相关|"""

    def faithfulness_score(llm_answer, retrieved_docs_content):
        """忠实度 = 答案中能在上下文中找到支撑的句子比例"""

    def answer_relevancy_score(question, llm_answer):

        """对所有测试用例运行完整评估"""

### 第 4 部分：评估最佳实践

**本节讲什么：**
- 建立基线（Baseline）的方法
- 控制变量原则（每次只改一个）
- 评估频率（开发/上线前/上线后）
- CI/CD 集成
- 人工评估的不可替代性

**6 条最佳实践：**

| # | 实践 | 说明 | 反例 |
|---|------|------|------|
| 1 | **建立基线** | 改任何参数前先跑一次评估，记录基准分数 | 直接改参数，不知道变好还是变差 |
| 2 | **每次只改一个变量** | 只改 Embedding，其他不变，对比前后分数 | 同时改 Embedding + 切片 + LLM |
| 3 | **评估频率** | 开发阶段：每次重大修改后；上线前：完整评估；上线后：定期检测衰减 | 只在上线前测一次 |
| 4 | **自动化** | 评估脚本集成到 CI/CD，每次 PR 自动跑评估 | 手动跑评估，容易遗漏 |
| 5 | **人工评估不可替代** | 定期人工抽查 10-20 条回答，关注流畅度/有用性/毒性 | 完全依赖自动化指标 |
| 6 | **持续更新评估集** | 知识库更新→评估集也更新；收集用户真实问题加入评估集 | 评估集一成不变 |

## 🧠 核心知识点总结

### 四大指标关系图

```
         RAG 评估
        /        \
   检索质量      生成质量
   /      \      /      \
精确率   召回率  忠实度   相关性
  ↑        ↑      ↑        ↑
检索到的  找到的  不编造   答到点
都相关    没遗漏  有依据   子上
```

### 评估指标速查表

| 指标 | 评估对象 | 满分含义 | 常见低分原因 |
|------|---------|---------|-------------|
| Context Precision | 检索器 | 所有检索结果都相关 | 检索方法不当、阈值太低 |
| Context Recall | 检索器 | 没有遗漏相关内容 | 知识库不完整、切片太粗 |
| Faithfulness | LLM 生成 | 回答没有编造任何内容 | LLM 幻觉、Prompt 不够严格 |
| Answer Relevancy | LLM 生成 | 完全回答了问题 | 检索到的内容不相关 |

## ⚠️ 常见问题

### Q: 教学代码中的评估和 ragas 库有什么区别？
A: 教学代码实现了简化的指标计算（基于关键词匹配），用于理解概念。ragas 库使用 LLM 来评估（如让 LLM 判断答案是否基于上下文），更准确但需要 API 调用。教学版是"理解原理"，ragas 是"生产使用"。

### Q: 评估数据集从哪里来？
A: 初期手动创建 20-30 条覆盖核心功能的测试用例。上线后收集用户真实问题，逐步扩充到 100+ 条。也可以让 LLM 基于文档自动生成问答对作为补充。

### Q: 精确率和召回率哪个更重要？
A: 取决于场景。客服问答需要高召回率（不能漏掉关键信息）；法律/医疗需要高精确率（不能给不相关的信息）。通用 RAG 建议两者都追求，通过混合检索 + Rerank 实现。

### Q: 忠实度低说明什么？
A: 说明 LLM 在"编造"上下文中没有的信息。可能原因：① LLM 太"自由"，需要 Prompt 中加强约束（如"仅基于以下信息回答"）；② 检索到的内容确实不包含答案，LLM 试图用自己的知识补全。

### Q: 多久评估一次比较合适？
A: 开发阶段每次重大修改后（换模型、改参数、加数据）；上线前完整评估 + 人工抽查；上线后每周或每月定期评估，检测效果衰减（知识库过期等问题）。
